# 09 — Дизайн пакета S6 (`users-me`)

## Задача

Спроектировать шестой (последний по бэклогу) слайс сервиса — `GET /v1/users/me` — по дисциплине `program-design.skill`. На входе: контракт OpenAPI (200 / 401 / 503, без 507 — read-only), карта режимов отказа в README, один Gherkin-сценарий happy path (`Возвращает данные пользователя из токена`). На выходе: пакет проектной документации в `docs/design/passkey-demo/`, готовый к подхвату sonnet'ом.

## Ключевые промпты

Записаны дословно, в порядке появления в сессии.

1. **Оператор:** «Загрузи контекст. Бери задачу проектировать след слайс по бэклогу».
   *Агент* загрузил `AGENTS.md`, `CLAUDE.md`, `backlog.md`, скилл `program-design`, карточку S5 (свежий образец стиля), `messages.md`, `contracts-graph.md`, `infrastructure.md`, OpenAPI на `/v1/users/me`, Gherkin `users.feature`. Сформулировал четыре развилки (длина пайпа, маппинг `ErrUserNotFound`, scope Gherkin, аддитивные расширения S2) и предложил дефолты — стандартный порядок узлов (`VerifyAccessToken` → `Store.LoadUser` → `buildResponse`).

2. **Оператор:** «сначала найди пользователя по id потом валидировать токен».
   *Агент* не стал молча принимать перестановку. Зафиксировал три возможных прочтения и попросил уточнить:
   - (1) распарсить `Subject` без verify → `Store.LoadUser` → проверить подпись/exp/iss;
   - (2) `VerifyAccessToken` (атомарный) → `Store.LoadUser` (порядок «I/O *после* верификации» уже такой);
   - (3) что-то третье — например, перенести проверку `exp`/`iss` после загрузки.
   В прочтении (1) обозначил три trade-off: (а) leak existence через timing на любом UUID, (б) ломается подправило «подтип, не guard» (`User` на руках до verify — нужен промежуточный подтип), (в) код S6 расходится с S5. В текущей реализации сервер-ключ Ed25519 глобальный и не привязан к user, поэтому «загрузить user, чтобы взять его ключ» не применимо.

3. **Оператор:** «в devlog запиши наши промпты».
   *Агент* создал ветку `feat/design-users-me` (правило `feedback_no_direct_main_commits`) и завёл этот файл с дословной записью промптов; раздел «Принятое решение» заполнится после того, как развилка по порядку узлов будет закрыта.

## Контекст и ограничения

- S5 реализован и смержен (PR #36). Все аддитивные расширения S2 (`VerifyAccessToken`, `AuthenticatedUserID`, `VerifyAccessTokenInput`, `ErrAccessTokenInvalid`) уже экспортированы — S6 их импортирует без новых экспортов.
- S3 уже экспортировал из S2 рехидраторы `UserFromRow` и `UserIDFromString` — S6 готов использовать как есть.
- S6 — read-only эндпоинт. OpenAPI декларирует только 200 / 401 / 503 (нет 507 — read SQLite не различает `disk_full`).
- В Gherkin (`users.feature`) сейчас только happy-path. По аналогии с S5 — допустимо сознательно ограничить scope (маппинги 401/503 фиксируются в карточке как часть OpenAPI-контракта).
- Новых миграций не требуется — таблица `users` существует с S2 (`0002_users.sql`).

## Принятое решение

**Пайп головного модуля — 4 узла:** `NewUsersMeCommand` → `VerifyAccessToken` → `Store.LoadUser` → `buildResponse`. В диапазоне 5–10 не помещается (сабфлор скилла), как и S5 (3 узла); обоснование в `slices/06-users-me.md` секция «Решения по дизайну → Длина пайпа: 4 узла».

**Стандартный порядок «verify → load».** В обсуждении промпта 2 рассмотрено и отклонено инвертированное прочтение «extract Subject без verify → `Store.LoadUser` → потом подпись» из-за трёх минусов: timing-leak (БД дёргается на любых правильно структурированных JWT), ломается «подтип, не guard» (нужен лишний промежуточный тип), расхождение с шаблоном S5. Сервер-ключ Ed25519 глобальный — мотивации для инверсии (per-user keypair) нет.

**Маппинг `ErrUserNotFound` → 500 `INTERNAL_ERROR`.** После успешной верификации JWT `User` отсутствует в БД — это аномалия согласованности, не клиентская ошибка. На текущей кодовой базе случай невозможен (нет операции удаления `users`); зафиксирован для будущего admin-эндпоинта `DELETE /users/{id}`. Альтернативы 401 (скрывает баг под обычной экспирацией) и 404 (не декларирован OpenAPI) — отклонены.

**`ErrMissingBearer` — локальный sentinel в пакете `users_me`**, не импорт из `sessions_logout` (S5). Структурно архитектура slice-by-slice не предполагает прямой зависимости `S6 → S5` (оба — листья, расширяющие S2). Дублирование одной строки `errors.New(...)` — приемлемая цена локальности; вынос в общий `internal/auth/` модуль — преждевременно на двух эндпоинтах.

**Аддитивных расширений S1/S2/S3/S4/S5 не требуется.** S6 — первый слайс, который собирается из существующих публичных API без новых экспортов в других пакетах: `VerifyAccessToken`/`AuthenticatedUserID`/`VerifyAccessTokenInput`/`ErrAccessTokenInvalid` уже введены в S5; `UserFromRow`/`UserIDFromString` — в S3. Это побочный эффект того, что S5 уже сделал всю работу по Bearer-auth, а S3 — по рехидрации `users`-строк.

**Без новых миграций.** Таблица `users` существует с S2 (`0002_users.sql`); один SELECT по PK не требует расширения схемы или дополнительных индексов.

**Маппинг `ErrDiskFull` не вводится.** Read-only эндпоинт; SELECT не возвращает `SQLITE_FULL`. OpenAPI на S6 декларирует только 200/401/503 (без 507) — соответствует.

**Юнит-тесты — 3 штуки** (минимум за всю реализацию проекта): 2 на `NewUsersMeCommand` (happy + empty Bearer) + 1 на `buildResponse` (чистый маппинг доменной сущности в DTO). `VerifyAccessToken` уже посчитан в S5 (6 юнитов в `internal/slice/registrations_finish/`), не пересчитывается. Голова, I/O и ингресс-адаптер — трубы, юнитами не покрываются.

## Отклонённые варианты

- **Инверсия порядка «load по id → потом verify».** Минусы: timing-leak existence через БД, ломает «подтип, не guard» (нужен лишний `UserBeforeVerify`-подтип), расходится с S5. Глобальный server-key делает мотивацию (per-user keypair) неприменимой. См. диалог промпта 2.
- **Маппинг `ErrUserNotFound` → 401 `UNAUTHORIZED`.** Удобнее клиенту (логинится заново), но скрывает consistency-баг под обычной экспирацией. Оператор не получит сигнал в 5xx-алертах.
- **Маппинг `ErrUserNotFound` → 404 `NOT_FOUND`.** OpenAPI не декларирует 404 на `/v1/users/me` — введение нового кода ответа без правки спеки нарушает API-First (`AGENTS.md §2`).
- **Импорт `ErrMissingBearer` из `sessions_logout` (S5).** Вводит формальную зависимость S6 → S5, которой архитектурно нет. Цена — структурная сложность ради экономии одной строки.
- **Вынос `ErrMissingBearer` в S2** (рядом с `ErrAccessTokenInvalid`). Аддитивный рефакторинг S5, но расширяет scope handoff'а; преждевременно на двух эндпоинтах.
- **Дополнительный индекс на `users.id` или композитный**. PK на `id` (TEXT PRIMARY KEY) уже даёт O(log n) lookup; cardinality маленькая, оптимизация не нужна.
- **Сложить `parseAccessToken` и `NewAuthenticatedUserID` в два отдельных узла.** Дробление не даёт нового инварианта в типе и удлиняет пайп ради счёта (тот же аргумент, что в S5 для `VerifyAccessToken`).
- **Вынести `buildResponse` в адаптер.** Формально допустимо (адаптер уже формирует response), но `buildResponse` — это маппинг доменной сущности в DTO, чистая функция логики. Держать узлом графа делает явным, какие именно поля попадают в JSON (важно для будущих расширений профиля — `email`, `created_at`, флаги).

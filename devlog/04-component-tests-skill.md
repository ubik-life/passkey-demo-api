## Решения

**Зачем скилл, а не правило в AGENTS.md.** Жанровое разделение: AGENTS.md — обзор правил для агента, скиллы — детерминированные процедуры. AGENTS.md уже большой; раздувать §6 пятью шагами алгоритма и формулами означает мешать жанры. Скилл — отдельный артефакт по открытому стандарту SKILL.md от Anthropic.

**Расположение `skills/` на верхнем уровне.** По конвенции Anthropic в публичных репозиториях skills и согласно статье 04. Не в `docs/` — это исполняемая инструкция для агента, не пользовательская документация.

**«Карта режимов отказа» в README, не в AsyncAPI.** AsyncAPI описывает контракт сообщения между сервисами. Отказ инфраструктуры (БД недоступна) — не часть контракта между сервисами, а поведение конкретного сервиса при отказе **его** инфраструктуры. Видимый всем потребителям документ — README.

**Один режим отказа SQLite, не два.** В OpenAPI один `error.code: db_unavailable` под 503 — сервис не различает таймаут и недоступность. Это проектное решение, фиксирующее число сценариев: 6 эндпоинтов + 1 режим = 7 тестов.

**Терминология «отказ интеграции», не «отказ БД».** По итогам уточнения автора: формулировка должна совпадать с языком скилла («режим отказа интеграции i» из формулы) и оставлять место будущим режимам без переписывания. В backlog.md и README используется «отказ интеграции с БД».

**Один коммит на весь шаг.** Изменения связаны логически: SKILL.md без обновлённых AGENTS.md и README не работает (скилл ссылается на «Карту режимов отказа», AGENTS.md ссылается на скилл). Дробить на отдельные коммиты — создавать промежуточные неконсистентные состояния.

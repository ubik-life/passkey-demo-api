## Найденные разрывы

При первом чтении `api-specification/openapi.yaml`:

- **5xx-ответов нет ни на одном эндпоинте.** Статья `ai-ubik-04` декларировала, что у passkey-demo-api есть `503 db_unavailable` на всех шести — на самом деле в спеке этого не было. Контракт был мёртв.

## Анализ интеграций passkey-demo-api

Применил правило: **интеграция = выход за процесс сервиса** (сеть / ФС / IPC).

| Кандидат | Вердикт |
|---|---|
| SQLite | **интеграция** (драйвер БД, файл на диске) |
| WebAuthn-библиотека | модуль (Go-пакет в памяти) |
| Ed25519, JWT | модуль (чистые криптофункции) |
| Файл с ключом Ed25519 | модуль (генерируется при старте, не персистится) |
| HTTP inbound | поверхность сервиса, не интеграция |

**Интеграция одна — SQLite.**

## Хронология решения по числу режимов

| Раунд | Решение | Тестов | Триггер |
|---|---|---|---|
| 1 | 1 режим (`db_unavailable`) | 7 | Слепое следование статье `ai-ubik-04` |
| 2 | 3 режима (`db_unavailable`, `db_locked`, `db_disk_full`) | 9 | Критика: «всё в 503 — не разрулить ops/SRE» |
| 3 | 2 режима (`db_locked`, `db_disk_full`) | **8** | Критический вопрос: «реально ли поставить каждый режим в тесте?» — `db_unavailable` для embedded SQLite нереализуем чисто |

### Почему `db_unavailable` ушёл

Для встроенной SQLite режим оказался синтетическим:

- **Старт**: SUT панически падает при отказе БД (`AGENTS.md`), сервис не поднимется → 503 не вернётся вообще.
- **Mid-test**: открытый файловый дескриптор переживает `chmod 000`/`unlink` по POSIX-семантике — сервис продолжит писать, тест не поймает отказ.
- **Альтернативы** (lazy-init адаптера + `MaxIdleConns=0`, unmount тома, перезапуск SUT с битым путём) либо требуют архитектурной правки сервиса, либо платформенно-зависимы, либо превращают runtime-проверку в cold-start.

**Принцип, выкристаллизованный:** в контракт идут только **реально воспроизводимые в тестах** режимы, не теоретически возможные. `db_unavailable` вернётся при миграции на сетевую БД (Postgres) — там он естественный.

### Финальная карта

| Режим | HTTP | Заголовки | Клиент | Оператор |
|---|---|---|---|---|
| `db_locked` | 503 | `Retry-After: 1` | Ретраить с backoff | — |
| `db_disk_full` | 507 | — | **Не** ретраить | Эскалация |

`507` — только на write-эндпоинтах (всё, кроме `GET /v1/users/me`). Constraint violation (UNIQUE handle) — доменная ошибка, юнит-тест, не компонентный.

**Правило различимости в SKILL:** режим — отдельный, если различается хотя бы одно из четырёх — HTTP-код, заголовки, действие клиента, действие оператора. Вытесняет субъективное «нужно ли различать».

## Шаг 2.0 — шаблон компонентных тестов на Go (промпты 5–6)

Автор показал JVM-аналог: [`codemonstersteam/mq-rest-sync-adapter/component-tests`](https://github.com/codemonstersteam/mq-rest-sync-adapter/tree/main/component-tests). Cucumber + Gradle + Docker Compose + WireMock + IBM MQ. Попросил структурный аналог на Go.

**Ключевое отличие нашего случая:** SQLite встроенная — нет внешних контейнеров-стабов. «Стабы» сводятся к параметризации SQLite-mount через env-профили compose-файла.

**Решение по изоляции (промпт 6):** раннер в Docker, не с хоста. `Dockerfile.runtime` обязательный, `scripts/run-tests.sh` использует `docker compose run runner`. Снимает «у меня работает, в CI падает».

Шаг 2.0 разложен на 14 тикетов: Dockerfile раннера, compose-инфраструктура, базовые степы (HTTP / WebAuthn / DB-failure / auth), smoke-проверка, devlog. Выполняет opus или человек, не sonnet.

Промпт-шаблон для sonnet в Шаге 2 обновлён: теперь требует читать `component-tests/README.md`, `HOW-TO.md` и список степов в `steps/`, использовать существующие формулировки.

## Решения

**Принцип «контракт — то, что воспроизводится в тестах».** Финальный — после раундов 2 и 3. Не уходим в полноту теоретического перечня. Если режим нельзя поставить в реальном тесте, он не идёт в контракт. Это снимает класс «спецификация описывает то, что мы не можем доказать».

**Шаг 0 в SKILL вместо отдельного скилла.** Проектирование режимов и раннера — часть процедуры компонентных тестов. Без них Шаг 1 проваливается.

**Правило различимости.** Четыре оси (HTTP-код, заголовки, действие клиента, действие оператора) превращают субъективное «нужно ли разделять?» в проверяемое «различается ли что-то из четырёх?».

**507 для `db_disk_full`, не 503.** Семантически правильнее: retry бесполезен, отдельный код позволяет alerting'у среагировать без чтения тела.

**Шаг 2.0 предшествует Шагу 2.** Sonnet не может писать `.feature` без раннера и базовых степов. Шаблон делает opus/человек.

**Раннер в контейнере, не с хоста.** Полная изоляция, единое окружение в CI и локально, внутренняя docker-сеть.

**`db_unavailable` отложен (а не «оставлен на всякий случай в спеке»).** Если режим невоспроизводим, его в спеке быть не должно — иначе клиент будет полагаться на обещание, которое сервис не способен выполнить детерминированно.

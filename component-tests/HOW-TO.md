# HOW-TO — методология компонентных тестов passkey-demo-api

Этот документ — короткий ответ на вопрос «зачем именно так». Тактика —
в [`README.md`](README.md), пошаговая процедура генерации `.feature` —
в [`../skills/component-tests/SKILL.md`](../skills/component-tests/SKILL.md).

## Что такое компонентный тест

Тест — это **исполняемая спецификация чёрного ящика**. Не охота за багами,
не покрытие кода, не smoke-проверка сборки. Один артефакт читают четверо:

* аналитик — формализует требования;
* разработчик — пишет код под контракт;
* вайб-кодер / ИИ-агент — генерирует код по спецификации;
* машина — гоняет сценарии и сверяет поведение.

Поэтому `.feature` пишется на русском Gherkin: язык предметной области,
не код, не моки.

Подробно: [статья «Сколько компонентных тестов нужно сервису»](https://codemonsters.team/blog/2026/04/25/testing-mythology-component-tests/).

## Сколько тестов нужно

```
N_тестов = N_эндпоинтов_API + Σ режимы отказа интеграции i
```

Для passkey-demo-api: **6 эндпоинтов + 2 режима SQLite = 8 сценариев**.
Не больше, не меньше. Каждый сценарий — одно утверждение спецификации.

Подробно — в `../README.md` раздел «Карта режимов отказа» и в
[статье «Компонентные тесты на практике»](https://codemonsters.team/blog/2026/04/26/component-tests-skill-passkey-demo/).

## Почему раннер в Docker

Полная изоляция окружения раннер ↔ SUT:

* окружение в CI и локально — одно и то же;
* раннер не зависит от Go-toolchain хоста;
* сеть между раннером и сервисом — внутренняя docker-сеть, не loopback;
* отказ воспроизводится одинаково везде.

`go test ./steps/...` с хоста не запускается **никогда**. Только
`./scripts/run-tests.sh`.

## Почему `mattn/go-sqlite3`, не `modernc.org/sqlite`

`mattn` — стандарт де-факто, требует CGO, имеет полную функциональность
SQLite. Сервис собирается с `CGO_ENABLED=1`, статически линкуется
(см. `compose/service.Dockerfile`), runtime — `alpine:3` с wget для
HEALTHCHECK.

`modernc.org/sqlite` (чистый Go) был бы быстрее в CI (мелкие образы),
но для обучающего материала важнее показать стандартный стек, который
встретится в любом проде.

## Что НЕ проверяем компонентным тестом

* **сборку и запуск** — это `go build` и compose-healthcheck, а не сценарий;
* **валидацию полей** (длина handle, формат email, диапазон возраста) —
  это юнит-уровень модуля валидации;
* **бизнес-логику и ветвления** — это юниты внутри сервиса;
* **constraint violations** (UNIQUE handle) — доменная ошибка, юнит;
* **режимы отказа, которые мы не можем воспроизвести в тесте** —
  для встроенной SQLite это `db_unavailable` (нет «сетевой недоступности»),
  он вернётся при миграции на сетевую БД.

Принцип: **в спеку идут только реально воспроизводимые в тестах режимы**.

## TDD-red фаза

`.feature`-файлы пишутся **до кода сервиса** (см. AGENTS.md §9). Сейчас
сервис — это placeholder из `cmd/api/main.go`, отдающий 501 на всё, кроме
`/health`. Все сценарии Шага 2 будут красные — это ожидаемое состояние.

В Шаге 3 placeholder постепенно вытесняется реальной логикой модулей,
и сценарии становятся зелёными по одному.

## Откуда брать степы

Готовый набор — в `steps/`. Не выдумывай новые формулировки, если есть
существующая. Если для сценария не хватает степа — останови работу и
сообщи, добавим в Stage B.

Список с регулярками — в `README.md` раздел «Доступные степы».

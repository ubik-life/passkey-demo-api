# Devlog — passkey-demo (реализация)

## S1 — HTTP POST /v1/registrations (2026-05-01)

**Что сделано:** Реализован слайс `registrations-start` по дереву модулей из карточки S1. Заменён placeholder `cmd/api/main.go` на реальный сервер с chi-роутером, goose-миграциями, JSON-логгером (slog) и graceful shutdown. Создана вся инфраструктура: `internal/app/`, `internal/db/`, `internal/clock/`.

**Решения, принятые по ходу:**

- **Env-переменные.** Infrastructure.md называл `PASSKEY_LISTEN_ADDR`/`PASSKEY_DB_PATH`, но compose-инфра (Шаг 2.0) уже использовала `SERVICE_ADDR`/`SQLITE_PATH`. Решение: использовать `SERVICE_ADDR`/`SQLITE_PATH` (как в compose), `PASSKEY_RP_NAME`/`PASSKEY_RP_ID`/`PASSKEY_CHALLENGE_TTL` — со значениями по умолчанию для demo.

- **Мокирование I/O в юнит-тестах.** Spec требовала "тестируется с моками I/O" для головного модуля, но `Deps` по design-doc имел `*sql.DB`. Добавлен `Persist func(RegistrationSession) error` в `Deps`, который устанавливается через `NewDeps` (замыкание над `*sql.DB`). Юниты подменяют `Persist` напрямую без SQLite-процесса.

- **Права на volume.** Docker-volume `/var/lib/passkey` монтируется как root:root. Исправление: в Dockerfile добавлен `mkdir -p /var/lib/passkey && chown app:app` перед `USER app:app`, плюс `os.MkdirAll` в `db.Open`.

- **DSN SQLite.** Использован `file:` URI-префикс для поддержки query-параметров (`_journal=WAL`, `_busy_timeout=5000`).

- **Smoke test.** Старый `smoke.feature` проверял 501 от placeholder — обновлён на `GET /health → 200`, что является постоянным контрактом сервиса.

**Тесты:** юнитов 11 (4+2+1+1+1+1+1), 100% покрытие конструкторов и модулей логики. Головной модуль юнитами не покрывается — склейка уже протестированных частей. Компонентные сценарии: `Создание challenge регистрации` зелёный, `Smoke` зелёный. Остальные 7 сценариев красные ожидаемо (слайсы 2–6 не реализованы), все падают на фазе 2, не на фазе 1.

**Стоимость сессии:** $4.07 (claude-sonnet-4-6). Время API: 19m 12s, wall: 42m 19s. Изменений: +926 / -73 строк.

## S2 — HTTP POST /v1/registrations/{id}/attestation (2026-05-01)

**Что сделано:** Реализован слайс `registrations-finish` — фаза 2 регистрации. WebAuthn attestation verification (go-webauthn/webauthn), создание User+Credential, выдача JWT Ed25519 (access token) + opaque refresh token (sha256-хеш в БД). 3 новые миграции (users, credentials, refresh_tokens), расширения S1 (ChallengeFromBytes, RegistrationIDFromString, RegistrationSessionFromRow, RPConfig.Origin). Подтип FreshRegistrationSession (сессия не истекла). HTTP-обработчик со всеми кодами ошибок по контракту.

**Решения, принятые по ходу:**

- **API go-webauthn Verify.** Сигнатура `ParsedCredentialCreationData.Verify` в v0.17.0 имеет 11 параметров и отличается от старых примеров. Найдена по исходнику в GOMODCACHE. Параметр `credParams` обязателен — при `nil` верификация падает с "algorithm not supported".

- **virtualwebauthn API.** Пакет не имеет методов `auth.NewCredential()`, `auth.GetAttestationFor()`. Правильный вызов: `vwa.NewCredential(vwa.KeyTypeEC2)` (функция), `vwa.CreateAttestationResponse(rp, auth, cred, options)` (функция, не метод).

- **Честные юниты с virtualwebauthn.** Вместо моков WebAuthn-слоя используется virtualwebauthn прямо в юнит-тестах `logic_test.go` + in-memory SQLite в тестах головного модуля. Никаких stub-интерфейсов в Deps.

- **Изоляция компонентных сценариев.** SQLite-файл шарится между всеми сценариями (общий Docker volume). Без очистки: "bob" создаётся в "Завершение регистрации", затем background-шаг "sessions" пытается создать "bob" снова → 422 HANDLE_TAKEN. Исправление: `beforeScenario` открывает SQLite и делает DELETE FROM всех 4 таблиц + удаляет junk.bin от disk-full сценария.

- **AppConfig.JWT.** В S1 не было JWT-конфигурации. Добавлен `rf.JWTConfig` в `AppConfig` с тремя полями (AccessTTL, RefreshTTL, Issuer) и env-переменными PASSKEY_JWT_ACCESS_TTL/REFRESH_TTL/ISSUER. Signer (Ed25519 пара) генерируется при старте и не персистируется.

**Тесты:** юнитов 23 (3+2+1+1+1 domain + 4+2+1+1+7 logic/head). Компонентные сценарии: `Завершение регистрации` зелёный. Background-шаг "пользователь зарегистрирован и залогинен" зелёный. `Диск переполнен` красный ожидаемо (нужен профиль disk-full с tmpfs). Слайсы S3–S6 красные ожидаемо (не реализованы).

**Стоимость сессии:** $6.27. Время API: 21m 57s, wall: 2h 0m 20s. Изменений: +1462 / -170 строк.
Токены claude-sonnet-4-6: 8.3k input, 80.3k output, 13.4m cache read, 270.1k cache write.
Токены claude-haiku-4-5: 353 input, 15 output (вспомогательные вызовы, $0.0004).

## Рефакторинг S2 — удаление head-тестов и Rand из Deps (2026-05-02)

**Промпт:** «посмотри на юнит тесты — они покрывают head? используются моки?»

**Что найдено:** 7 тестов `TestProcessRegistrationFinish_*` в `logic_test.go` покрывали головной модуль с реальным in-memory SQLite. Плюс заглушка `alwaysFailReader` — единственная причина которой было поле `Rand io.Reader` в `Deps`.

**Что сделано:** Удалены 7 head-тестов + `alwaysFailReader`. `Rand io.Reader` убран из `Deps` — `crypto/rand.Reader` захардкожен внутри `generateTokenPair`. Из `wire.go` и `main.go` убран параметр `rnd io.Reader`.

**Решение:** `Rand io.Reader` в `Deps` — классический признак попытки юнит-тестировать головной модуль: поле нужно только чтобы подставить заглушку в тест. Правило: если поле в `Deps` нужно только ради тест-двойника — это не зависимость, это дыра. Захардкодить реальное значение внутри функции.

**Тесты:** 16 юнит-тестов (7 head-тестов удалены), все зелёные.

## Аудит скиллов — исправление систематических противоречий (2026-05-02)

**Промпт:** «посмотри критично на скилл проектирования, там нет ошибок, которые протекают в задачи по реализации модулей io?»

**Что найдено оператором:** Два противоречия между `program-design` и `program-implementation`:

1. **Скилл дизайна включал головной модуль в формулу юнит-тестов** — `N = 1 + Σ (ветки антецедента)` с фразой «головной модуль слайса». Opus честно следовал правилу и проектировал `TestProcessXxx_Happy`, `_SessionNotFound`, `_SessionExpired` и т.д. Sonnet так же честно реализовывал их. Потом приходилось удалять вручную — как только что.

2. **Скилл дизайна обосновывал IO-изоляцию через моки** — «каждый I/O-модуль оборачивается ровно одним mock'ом / fake'ом в юнит-тестах головного модуля». Это прямо провоцировало проектировать подменяемые поля в `Deps` (`Rand io.Reader`) — нарушение правила «без моков».

**Суть:** Ошибка жила не в коде, не в агенте — она жила в скиллах. Пока скилл дизайна предписывал одно, а скилл реализации запрещал это же — машина стабильно нарушала правило, потому что следовала более раннему артефакту (дизайну). Починка отдельного PR не убирала причину: следующий slice был бы написан так же.

**Что исправлено:**
- `program-design/SKILL.md`: убран head из формулы и таблицы юнит-тестов; добавлены разделы «Головной модуль — оркестратор-труба» и «Правило автономного IO-объекта / пустой трубы»; убрано упоминание mock/fake
- Дизайн-карточки S1 и S2: приведены в соответствие (head убран из таблиц, счёт 14→11 и 23→16)

**Урок для работы с машиной:** Повторяющаяся ошибка — это сигнал искать не в коде, а в артефактах, которые машина читает раньше кода. Скилл дизайна читается до скилла реализации, поэтому противоречие в скилле дизайна победит любое правило в скилле реализации. Мастерство с машиной — не «лучше объяснить агенту», а «убрать противоречие из источника».

**Стоимость сессии:** $4.61 (за вычетом S1 $4.07 + S2 $6.27 из накопленного $14.95). Время API: 6m 32s.
Токены claude-sonnet-4-6: 10.0k input, 171.2k output, 33.1m cache read, 618.6k cache write (накопленно).
Токены claude-haiku-4-5: 373 input, 5.6k output, 44.8k cache read, 45.8k cache write (накопленно, $0.09).

## Дизайн S3 sessions-start + унификация S1/S2 + чек-лист в скилле (2026-05-02)

**Что сделано:** Спроектирован slice S3 (`POST /v1/sessions`) — карточка `slices/03-sessions-start.md` (8 узлов, псевдокод, контракты, Gherkin-mapping, 6 юнит-тестов, DoD), дополнения `messages.md`/`contracts-graph.md`/`infrastructure.md` (миграция `0005_login_sessions.sql`, `Deps` слайса 3), тикет S3 и хендофф в `design/backlog.md`. Карточки S1/S2 переписаны под автономный I/O-объект `Store` (методы `Store.PersistRegistrationSession`, `Store.LoadRegistrationSession`, `Store.FinishRegistration`); сырой `*sql.DB` убран со стрелок графов и из `Deps` головных модулей в дизайне. Скилл `program-design` дополнен жёстким чек-листом `Dependencies:` в Шаге 5 (таблица запрещённых сырых клиентов) и пунктом в хендофф-чеклисте Шага 12. Тикет техдолга кода S1/S2 (`refactor/s1-s2-store`) зафиксирован в root `backlog.md`.

**Решения, принятые по ходу:**

- **Первый проход дизайна S3 содержал сырой `*sql.DB` в `Deps`** — нарушение Шага 6 + `feedback_io_autonomous_store`. Поймал оператор. Корневая причина — машина копировала форму из карточек S1/S2, которые сами были написаны до закрепления правила автономного IO-объекта (формулировка появилась в скилле во время аудита 2026-05-02, но карточки S1/S2 не были перепроектированы задним числом). Когда новый узел дизайна копирует форму из старого, старая ошибка тиражируется.

- **Развилка по фронту правок.** Точечное исправление только S3 оставляло три карточки в двух стилях — на S4 та же ошибка вернулась бы по тому же механизму. Решение: переписать S1/S2 карточки в одном PR с S3 (одна ветка, связанные правки) + добавить чек-лист в скилл.

- **Код S1/S2 не трогается в этой ветке.** Реализация (PR #17/#21) держит `Deps.DB *sql.DB` напрямую — это техдолг кода. По правилу «связанные правки — одна ветка»: ветка дизайна S3 не должна расширяться на рефакторинг реализаций. Зафиксирован отдельным тикетом в root backlog. Делать сразу после мержа этого PR, до начала дизайна S4.

- **Чек-лист в Шаге 5, а не только в ревью.** Правило существовало в Шаге 6, но срабатывало как «постусловие» в ревью. Машина при копировании формы из S2 не запускала это постусловие. Закрытие — механический чек-лист `Dependencies:` в Шаге 5 (момент заполнения контракта): таблица «Запрещено / Разрешено», алгоритм проверки. Это уже не творческое решение, а сверка по таблице — либо проходит, либо нет.

- **Один объект, два метода (read + write).** Альтернатива — два объекта (`UserCredentialReader`, `LoginSessionWriter`) — даёт больше типов без выигрыша в инкапсуляции. Один `Store` на слайс — соблюдает правило «один режим работы — один метод», объект инкапсулирует одну `*sql.DB`.

- **Каждый слайс имеет свой `Store`-тип.** `registrations_start.Store`, `registrations_finish.Store`, `sessions_start.Store` — независимые типы, общий `*sql.DB` под капотом. Vertical slice не делит код I/O между слайсами; общим остаётся только домен (`Handle`, `User`, `Credential`) и рехидраторы.

**Урок для работы с машиной (продолжение урока 2026-05-02):** Повторяющаяся ошибка живёт в артефактах, которые машина читает раньше кода. На S3 — в карточках S1/S2 (форма с сырым `*sql.DB`). Лечится не «больше внимания агенту», а двумя действиями: (а) убрать ошибку из источника-эталона (переписать карточки S1/S2), (б) поставить механическую сверку в самый ранний момент (чек-лист в Шаге 5, не Шаг 6). После этого следующий слайс физически не сможет повторить ошибку — её ловит чек-лист до того, как она попадёт в граф.

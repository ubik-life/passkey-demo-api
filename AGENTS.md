# AGENTS.md — Фрейм рациональной разработки с ИИ

Этот документ — рабочий контракт между разработчиком и ИИ-агентом.
Копируй и используй как системный контекст при проектировании сервисов.

---

## Структура постановки задачи

Каждую задачу описываем в следующем порядке в README.md:

1. **Постановка задачи** — что решаем, зачем, какой результат считается успехом
2. **Архитектура** — компоненты, границы сервиса, взаимодействие с внешними системами
3. **Описание работы программы** — сценарии выполнения, входные и выходные данные
4. **Схема иерархий модулей** — дерево модулей от верхнего уровня к нижнему
5. **Блок-схема алгоритма головного модуля** — управляющий поток главного entry point

---

## Правила разработки

### 1. REST API — лучшие практики

- Ресурсо-ориентированный дизайн: существительные в URL, глаголы — HTTP-методы
- Идемпотентность: `GET`, `PUT`, `DELETE` — всегда идемпотентны
- Коды ответов по смыслу: `201` при создании, `204` при удалении, `422` при валидационных ошибках
- Версионирование через URL: `/v1/...`
- Пагинация через `cursor` или `offset+limit` с явными метаданными

### 2. API First — OpenAPI / AsyncAPI 3.0

- Сначала контракт, потом код. Спецификация — единственный источник правды об интерфейсе
- Синхронные REST API описываем в **OpenAPI 3.x**
- Асинхронные события и очереди описываем в **AsyncAPI 3.0**
- Контракты хранятся в репозитории (`/api-specifiacation`), генерация кода — из спецификации
- Изменения API без изменения спецификации недопустимы

### 3. Формальная корректность программ

Источник: https://codemonsters.team/blog/2025/12/30/program-correctness/

**Суть:** каждый модуль проверяет допустимый диапазон входных и выходных значений.
Когда каждый модуль защищён — программа корректна в целом.

- Для каждого модуля определяем **предусловия** (P) и **постусловия** (Q)
- Валидируем входные данные до исполнения модуля
- Проверяем результат после исполнения модуля
- Используем инварианты циклов — явно формулируем и документируем
- Ветвление: каждая ветка рассматривается как отдельный путь с собственными условиями

### 4. Тесты — ровно сколько нужно

Не покрываем всё подряд — проверяем то, что важно:

| Категория | Что проверяем |
|-----------|---------------|
| Граничные значения | Минимум, максимум, ноль, пустая строка, nil |
| Корректная работа | Основной happy path сценарий |
| Ошибки модуля | Все задокументированные коды ошибок |

Цель: максимальная уверенность при минимальных ресурсах машины.

### 5. TDD — прагматичный цикл

```
Написать тест на модуль
      ↓
Написать модуль (тест красный → зелёный)
      ↓
Рефакторинг (тест остаётся зелёным)
      ↓
git checkout -b feature/...
      ↓
Коммит → пуш
      ↓
Обновить CLAUDE.md (контекст для следующей итерации)
      ↓
PR → merge в main
```

### 6. Компонентные тесты — спецификация чёрного ящика

- Компонентный тест = тест на сервис целиком через его публичный интерфейс
- Описываем сценарии на **Gherkin** (`Given / When / Then`)
- Gherkin-сценарии — это одновременно спецификация и автотест
- Тесты живут в `/component-tests`
- Сервис должен проходить компонентные тесты без знания о его внутренней реализации

### 7. Проектирование модулей

Источник: https://codemonsters.team/blog/2025/12/15/program-modules/

- **Одна функция** — один модуль. Единственная ответственность
- **Один вход, один выход** — никаких множественных точек входа/выхода
- **До 100 строк** исполняемого кода в модуле
- **Без истории вызовов** — модуль не помнит предыдущих вызовов
- **Детерминизм** — одинаковый вход всегда даёт одинаковый выход
- **Явная передача данных** — через параметры, не через глобальное состояние
- **Иерархия сверху вниз**: верхний модуль — сводка программы, нижний — конкретика
- **Принцип чёрного ящика**: вызывающему нужны только имя модуля и ожидаемый результат

### 8. Очереди и асинхронность

- **Очередь — основной транспорт** между сервисами там, где нет требования синхронного ответа
- Используем **at-least-once delivery** с идемпотентными обработчиками
- **Dead Letter Queue (DLQ)** — обязательна для каждой очереди
- Сообщения имеют явную схему (AsyncAPI 3.0 + JSON Schema / Avro / Protobuf)
- Потребитель не знает о производителе — только о контракте сообщения
- Логируем: `message_id`, `correlation_id`, статус обработки, время
- Retry-политика: экспоненциальный backoff с максимальным числом попыток

### 9. Флоу разработки сервиса

```
backlog.md (задача)
      ↓
Постановка задачи + Архитектура (AGENTS.md структура)
      ↓
API-контракт (OpenAPI / AsyncAPI) — merge в main
      ↓
Компонентные тесты на Gherkin (красные) — merge в main
      ↓
TDD-цикл по модулям:
  тест → модуль → зелёный → коммит → пуш → merge
      ↓
Компонентные тесты зелёные — сервис готов
      ↓
Обновить CLAUDE.md и backlog.md
```

### 10. Бэклог

- Ведём в файле **`backlog.md`** в корне репозитория
- Формат: задача, статус (`todo` / `in_progress` / `done`), краткое описание
- Экономим контекст: завершённые задачи переносим в раздел `## Done`
- CLAUDE.md — для технического контекста сессии; backlog.md — для планирования

### 11. Trunk Based Development

- Основная ветка — `main`, всегда в рабочем состоянии
- Ветки живут не дольше 1–2 дней
- Малые итерации: один коммит — одно логическое изменение
- Feature flags для незавершённых фич, которые нужно смержить досрочно
- Никаких долгоживущих feature-веток

### 12. Коммиты — лучшие практики

Формат: **Conventional Commits**

```
<type>(<scope>): <description>

[optional body]

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
```

| Тип | Когда |
|-----|-------|
| `feat` | новая функциональность |
| `fix` | исправление бага |
| `test` | тесты без изменения кода |
| `refactor` | рефакторинг без изменения поведения |
| `docs` | документация |
| `chore` | конфигурация, зависимости, CI |

- Один коммит — одно изменение
- Описание в повелительном наклонении: «добавить», «исправить», «удалить»
- Тело коммита — «почему», не «что» (что видно из diff)
- Не коммитим сломанный код в `main`

### 13. Observability — логирование, метрики, трейсинг

- **Structured logging**: все логи в JSON, уровни `debug / info / warn / error`
- Каждый запрос несёт `trace_id` и `span_id` — сквозной идентификатор через все сервисы
- Метрики по модели **RED**: Rate (RPS), Errors (%), Duration (p50/p95/p99)
- Логируем: входящие запросы, исходящие вызовы, ошибки, старт/стоп сервиса
- `correlation_id` передаётся в заголовках между сервисами и в сообщениях очереди
- Без observability нельзя понять, что происходит в проде

### 14. Обработка ошибок — единый стиль

- Явный тип ошибки — не просто `error`, а именованные коды как константы
- Ошибки не глотаем: логируем или пробрасываем с контекстом (`wrap` с описанием слоя)
- Единый формат error-ответа API:
  ```json
  { "code": "VALIDATION_ERROR", "message": "...", "details": [...] }
  ```
- HTTP-коды по смыслу: `400` — клиент виноват, `422` — данные не прошли валидацию, `500` — сервер виноват
- Граничный уровень (HTTP handler, queue consumer) — единственное место, где ошибка превращается в ответ

### 15. Миграции БД

- Только через миграционные файлы: **goose** (Go), **Flyway** (JVM), **migrate** (универсальный)
- Миграции живут в `/migrations`, пронумерованы и закоммичены в репозиторий
- Никаких ручных изменений схемы в проде
- Каждая миграция — один атомарный шаг; обратимые миграции (`up` / `down`)
- Миграции запускаются автоматически при старте сервиса или в CI перед тестами

### 16. Конфигурация и секреты

- Конфиг через **переменные окружения** (12-factor app): никакого хардкода
- Никаких секретов в коде или git-истории
- Обязательный **`.env.example`** с описанием всех переменных и их допустимых значений
- Секреты в проде — через Vault / Kubernetes Secrets / переменные CI/CD
- При старте сервис валидирует наличие обязательных переменных и падает явно, если что-то не задано

### 17. Контракт-тесты между сервисами ⚠️ TODO: автоматизация

Consumer-Driven Contract Testing — потребитель описывает ожидания к поставщику.
Если контракт нарушен — CI падает до прода.

**Текущий статус:** инструмент валидации спецификаций поставщика и потребителя требует доработки.
Референс: https://git.codemonsters.team/guides/mq-rest-sync-adapter/-/tree/main/contract-tests-v2?ref_type=heads

**Цель автоматизации:**
- Потребитель публикует ожидаемый контракт (схему запроса/ответа или сообщения)
- CI поставщика автоматически проверяет, что его спецификация удовлетворяет контракту потребителя
- Инструмент валидации сравнивает OpenAPI / AsyncAPI спецификации обеих сторон

**До завершения автоматизации:** контракт-тесты описываем вручную в `/contract-tests`

### 18. Работа с контекстом агента (CLAUDE.md)

`CLAUDE.md` — технический журнал текущей сессии разработки. Не дублирует AGENTS.md.

Что пишем в `CLAUDE.md`:
- Текущее состояние каждого модуля (`done` / `in_progress` / `todo`)
- Принятые архитектурные решения и их причины
- Открытые вопросы и блокеры
- Контекст, который нужен агенту при следующем запуске (что уже сделано, что ожидает)

Структура:
```
## Статус модулей
## Принятые решения
## Открытые вопросы
## Следующий шаг
```

Обновляем после каждого мержа в `main`.

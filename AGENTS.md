# AGENTS.md — Фрейм рациональной разработки с ИИ

Этот документ — рабочий контракт между разработчиком и ИИ-агентом.
Копируй и используй как системный контекст при проектировании сервисов.

---

## Документация как продукт — JTBD-сегментация

У документации репозитория четыре потребителя. Каждый нанимает её на свою работу.
Смешивать их в одном файле — гарантия того, что ни один не обслужен.

### Сегмент 1: Инженер-создатель (мейнтейнер)

**Job:** снизить стоимость онбординга до нуля — чтобы заниматься кодом, а не объяснениями.

**Документы:** `README.md` (верхняя часть) + `CONTRIBUTING.md` + `docs/adr/` + `docs/architecture.md`

Что нужно в README для создателя:
- Краткое описание продукта: какую задачу решает (1–2 предложения)
- Какие API есть + ссылка на OpenAPI-спеку
- **Pipe-описание каждого API** — как работает на пальцах:
  ```
  POST /v1/registrations
  | Принимаем handle от клиента
  | Генерируем challenge (32 байта, TTL 5 мин)
  | Сохраняем challenge в БД
  | Возвращаем challenge + WebAuthn options → 201
  ```
  Pipe показывает поток данных и зависимости. Таблица "метод → ресурс" отвечает на «что есть».
  Pipe отвечает на «как работает и где может сломаться».
- Как запустить и протестировать:
  - Команды запуска (`go run .` / `docker compose up`)
  - Ссылка на компонентные тесты (`/component-tests`) — исполняемая инструкция по проверке сервиса в изоляции
- Архитектурные решения — в `docs/adr/`, из README только ссылка
- Устройство программы — в `docs/architecture.md`, из README только ссылка

### Сегмент 2: Инженер-потребитель (другая команда)

**Job:** за 5 минут понять — решает ли эта штука мою проблему, и показать кратчайший путь к результату.

**Документ:** `README.md` — но потребитель читает его иначе, чем создатель.

Обязательные блоки:
- Что это и зачем (одно предложение — не абзац)
- Ссылка на платформу: «Часть платформы [Ubik](https://github.com/ubik-life/concept)» — чтобы понять из каких кусков состоит система целиком
- Что умеет / что НЕ умеет (границы ответственности сервиса)
- Стек (таблица: компонент → технология)
- API (таблица: метод, ресурс, действие)
- Запуск (минимальные шаги)

**Лестница вглубь:** потребитель может быть назначен на доработку.
README должен дать путь наращивания контекста:
```
README.md (что это, как запустить)
  → component-tests/ (как это работает снаружи)
    → docs/architecture.md (как устроена программа внутри)
      → docs/adr/ (почему сделано именно так)
        → CONTRIBUTING.md (как правильно менять)
```
Каждый уровень — отдельный документ. Потребитель останавливается на том уровне, который ему нужен.

### Сегмент 3: Менеджер / стейкхолдер

**Job:** за 3 минуты получить картину всей системы — из чего состоит, как связано.

**Документ:** README мета-репозитория (`ubik-life/concept`).

Обязательные блоки:
- Идея платформы (2–3 абзаца)
- Архитектура целевой системы (схема компонентов и их взаимодействия)
- Технологический стек (таблица: слой → технология → обоснование)
- Текущий шаг (что сейчас делаем, ссылка на активный сервис)

Каждый сервисный репозиторий содержит в первых строках README ссылку на concept:
```
> Часть платформы [Ubik](https://github.com/ubik-life/concept). Архитектура и концепция — там.
```

Менеджер не лезет в сервисные репозитории. Но если случайно попал — ссылка выведет его на нужный уровень.

**НЕ помещай в concept README:**
- API-эндпоинты и pipe-описания — это в README сервиса
- Архитектуру конкретного сервиса — это в `docs/architecture.md` сервиса
- Модель данных, команды запуска, структуру проекта — это в README сервиса
- Детали реализации любого уровня — concept описывает систему, не сервис

Для каждого шага разработки в concept README: одна строка «что умеет» + ссылка на репозиторий сервиса.

### Сегмент 4: ИИ-агент

**Job:** за минимум токенов загрузить контекст и быть готовым выполнить задачу.

**Документы:**
- `AGENTS.md` — неизменяемые правила разработки (этот файл). Копируется из шаблона, не редактируется в сервисе.
- `CLAUDE.md` — изменяемый технический журнал сессии. Состояние модулей, принятые решения, открытые вопросы, следующий шаг.
- `docs/architecture.md` — иерархия модулей и блок-схема entry point. Агент читает при старте, чтобы понять структуру программы.

Принципы написания для агента:
- Максимальная плотность информации на токен. Ноль воды.
- Один плоский файл. Агент не кликает — читает линейно.
- Структура через заголовки и таблицы, не через прозу.
- Явные запреты: «НЕ делай X», а не «старайся избегать X».

## Устройство программы — docs/architecture.md

Файл `docs/architecture.md` описывает внутреннее строение сервиса.
Потребители: инженер-создатель, инженер-потребитель (при доработке), ИИ-агент.

Обязательные разделы:

### Схема иерархий модулей

Дерево модулей от верхнего уровня к нижнему. Показывает кто кого вызывает.

```
main
├── server (HTTP-сервер, роутинг)
│   ├── handlers/registration (обработка регистрации)
│   │   ├── challenge (генерация challenge)
│   │   └── attestation (проверка attestation)
│   ├── handlers/session (обработка сессий)
│   │   ├── challenge (генерация challenge)
│   │   └── assertion (проверка assertion)
│   └── middleware/auth (проверка JWT)
├── store (работа с БД)
│   ├── users
│   ├── credentials
│   ├── sessions
│   └── challenges
└── tokens (генерация и валидация JWT)
```

Правила:
- Каждый узел — один модуль с одной ответственностью
- Стрелки только сверху вниз — нижний модуль НЕ знает о верхнем
- I/O-модули (store, HTTP) отделены от бизнес-логики
- **I/O-модуль автономен:** каждый I/O-объект инкапсулирует свою зависимость. Головной модуль знает только API объекта (методы), но не его зависимости. Имя объекта отражает тип интеграции: `Store` (БД), `Client` (HTTP), `Publisher`/`Consumer` (брокер). `Deps` слайса содержит объект, а не сырую зависимость (`*sql.DB`, `*http.Client`). Стиль Алана Кея: объект получает сообщение, сам знает, как его обработать.

### Блок-схема алгоритма головного модуля

Управляющий поток main entry point — что происходит при запуске программы.

```
main()
  │
  ├─ Загрузить конфигурацию из ENV
  │  └─ Нет обязательных переменных? → panic с явным сообщением
  │
  ├─ Открыть БД, запустить миграции
  │  └─ Ошибка? → panic
  │
  ├─ Инициализировать WebAuthn RP
  ├─ Сгенерировать или загрузить ключи подписи (Ed25519)
  ├─ Собрать роутер (Chi)
  │  ├─ /v1/registrations → registration handlers
  │  ├─ /v1/sessions → session handlers
  │  └─ /v1/users/me → auth middleware → user handler
  │
  └─ Запустить HTTP-сервер
     └─ Graceful shutdown по SIGTERM
```

Правила:
- Показываем только управляющий поток, не детали реализации
- Каждая ветка ошибки — явно: что происходит при сбое
- Обновляем при изменении entry point или добавлении новых доменов

## Правила разработки

### 1. REST API — лучшие практики

* Ресурсо-ориентированный дизайн: существительные в URL, глаголы — HTTP-методы
* Идемпотентность: `GET`, `PUT`, `DELETE` — всегда идемпотентны
* Коды ответов по смыслу: `201` при создании, `204` при удалении, `422` при валидационных ошибках
* Версионирование через URL: `/v1/...`
* Пагинация через `cursor` или `offset+limit` с явными метаданными

### 2. API First — OpenAPI / AsyncAPI 3.0

* Сначала контракт, потом код. Спецификация — единственный источник правды об интерфейсе
* Синхронные REST API описываем в **OpenAPI 3.x**
* Асинхронные события и очереди описываем в **AsyncAPI 3.0**
* Контракты хранятся в репозитории (`/api-specification`), генерация кода — из спецификации
* Изменения API без изменения спецификации недопустимы

### 3. Формальная корректность программ

Источник: <https://codemonsters.team/blog/2025/12/30/program-correctness/>

**Суть:** каждый модуль проверяет допустимый диапазон входных и выходных значений.
Когда каждый модуль защищён — программа корректна в целом.

* Для каждого модуля определяем **предусловия** (P) и **постусловия** (Q)
* Валидируем входные данные до исполнения модуля
* Проверяем результат после исполнения модуля
* Используем инварианты циклов — явно формулируем и документируем
* Ветвление: каждая ветка рассматривается как отдельный путь с собственными условиями

### 4. Тесты — ровно сколько нужно

Тест — это исполняемая спецификация поведения, не охота за багами и не покрытие кода.
Дейкстра: тестирование показывает наличие ошибок, но не доказывает их отсутствия.
Уверенность даёт проектирование, а не количество тестов.

**Юнит-тесты** проверяют модуль изолированно:

| Категория | Что проверяем |
| --- | --- |
| Граничные значения | Минимум, максимум, ноль, пустая строка, nil |
| Корректная работа | Основной happy path сценарий |
| Ошибки модуля | Все задокументированные коды ошибок |

**Компонентные тесты** проверяют сервис как чёрный ящик через API. Число сценариев — детерминированное:

```
N_тестов = N_эндпоинтов_API + Σ (число различимых режимов отказа интеграции i)
```

Контракт первичен: режимы отказа задокументированы в OpenAPI (5xx с `error.code`)
и в README в разделе «Карта режимов отказа», не выведены из кода адаптеров.

Полная процедура — в [`skills/component-tests/SKILL.md`](skills/component-tests/SKILL.md).

Цель: максимальная уверенность при минимальных ресурсах машины.

### 5. TDD — прагматичный цикл

```
Написать тест на модуль
      ↓
Написать модуль (тест красный → зелёный)
      ↓
Рефакторинг (тест остаётся зелёным)
      ↓
git checkout -b feature/...
      ↓
Коммит → пуш
      ↓
Обновить CLAUDE.md (контекст для следующей итерации)
      ↓
PR → merge в main
```

### 6. Компонентные тесты — спецификация чёрного ящика

* Тест = исполняемая спецификация чёрного ящика (как datasheet микросхемы)
* **Контракт первичен**: режимы отказа — в OpenAPI (5xx + `error.code`) и в README раздел «Карта режимов отказа», не из кода адаптера
* Формат: **Gherkin** (`Given / When / Then`) — один артефакт читают аналитик, разработчик, вайб-кодер и машина
* Один `.feature` файл на ресурс API; тесты живут в `/component-tests`
* Запуск: Docker Compose с реальными зависимостями, без моков
* Не дублируем юнит-уровень: валидация полей, бизнес-логика, ветвления — всё это уже доказано юнитами
* Полная процедура генерации — [`skills/component-tests/SKILL.md`](skills/component-tests/SKILL.md)

### 7. Проектирование модулей

Источник: <https://codemonsters.team/blog/2025/12/15/program-modules/>

* **Одна функция** — один модуль. Единственная ответственность
* **Один вход, один выход** — никаких множественных точек входа/выхода
* **До 20 строк** исполняемого кода в модуле
* **Без истории вызовов** — модуль не помнит предыдущих вызовов
* **Детерминизм** — одинаковый вход всегда даёт одинаковый выход
* **Явная передача данных** — через параметры, не через глобальное состояние
* **Модули I/O чётко отделены** от модулей с бизнес-логикой
* **Иерархия сверху вниз**: верхний модуль — сводка программы, нижний — конкретика
* **Принцип чёрного ящика**: вызывающему нужны только имя модуля и ожидаемый результат

### 8. Очереди и асинхронность

* **Очередь — основной транспорт** между сервисами там, где нет требования синхронного ответа
* Используем **at-least-once delivery** с идемпотентными обработчиками
* **Dead Letter Queue (DLQ)** — обязательна для каждой очереди
* Сообщения имеют явную схему (AsyncAPI 3.0 + JSON Schema / Avro / Protobuf)
* Потребитель не знает о производителе — только о контракте сообщения
* Логируем: `message_id`, `correlation_id`, статус обработки, время
* Retry-политика: экспоненциальный backoff с максимальным числом попыток

### 9. Флоу разработки сервиса

```
backlog.md (задача)
      ↓
API-контракт (OpenAPI / AsyncAPI) — merge в main
      ↓
README.md — заполнить по разделу «Структура постановки задачи в README» — merge в main
      ↓
docs/architecture.md — иерархия модулей и блок-схема entry point — merge в main
      ↓
Компонентные тесты на Gherkin (красные) — merge в main
      ↓
TDD-цикл по модулям:
  тест → модуль → зелёный → коммит → пуш → merge
      ↓
Компонентные тесты зелёные — сервис готов
      ↓
Обновить CLAUDE.md, backlog.md, README.md, docs/architecture.md
```

### 10. Бэклог

* Ведём в файле **`backlog.md`** в корне репозитория
* Формат: задача, статус (`todo` / `in_progress` / `done`), краткое описание
* Экономим контекст: завершённые задачи переносим в раздел `## Done`
* CLAUDE.md — для технического контекста сессии; backlog.md — для планирования
* **Обновляем после каждого тикета, не одним батчем в конце PR.** Закрыл тикет — сразу ставь `[x]` в `backlog.md` и актуализируй формулировки, если по ходу работы решение поменялось (например, ожидаемый HTTP-код в smoke-тесте). Backlog.md — единственный источник истины о статусе; разрыв между ним и реальностью ломает план для следующей сессии и для других агентов

### 11. Trunk Based Development

* Основная ветка — `main`, всегда в рабочем состоянии
* Ветки живут не дольше 1–2 дней
* Малые итерации: один коммит — одно логическое изменение
* Feature flags для незавершённых фич, которые нужно смержить досрочно
* Никаких долгоживущих feature-веток

### 12. Коммиты — лучшие практики

Формат: **Conventional Commits**

```
<type>(<scope>): <description>

[optional body]

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
```

| Тип | Когда |
| --- | --- |
| `feat` | новая функциональность |
| `fix` | исправление бага |
| `test` | тесты без изменения кода |
| `refactor` | рефакторинг без изменения поведения |
| `docs` | документация |
| `chore` | конфигурация, зависимости, CI |

* Один коммит — одно изменение
* Описание в повелительном наклонении: «добавить», «исправить», «удалить»
* Тело коммита — «почему», не «что» (что видно из diff)
* Не коммитим сломанный код в `main`

### 13. Observability — логирование, метрики, трейсинг

* **Structured logging**: все логи в JSON, уровни `debug / info / warn / error`
* Каждый запрос несёт `trace_id` и `span_id` — сквозной идентификатор через все сервисы
* Метрики по модели **RED**: Rate (RPS), Errors (%), Duration (p50/p95/p99)
* Логируем: входящие запросы, исходящие вызовы, ошибки, старт/стоп сервиса
* `correlation_id` передаётся в заголовках между сервисами и в сообщениях очереди

### 14. Обработка ошибок — единый стиль

* Явный тип ошибки — не просто `error`, а именованные коды как константы
* Ошибки не глотаем: логируем или пробрасываем с контекстом
* Единый формат error-ответа API:

  ```json
  { "code": "VALIDATION_ERROR", "message": "...", "details": [...] }
  ```
* HTTP-коды по смыслу: `400` — клиент виноват, `422` — данные не прошли валидацию, `500` — сервер виноват
* Граничный уровень (HTTP handler, queue consumer) — единственное место, где ошибка превращается в ответ

### 15. Миграции БД

* Только через миграционные файлы: **goose** (Go), **Flyway** или **Liquibase** (JVM)
* Миграции живут в `/migrations`, пронумерованы и закоммичены в репозиторий
* Только обратносовместимые изменения схемы — предыдущая версия сервиса должна работать при изменениях
* Никаких ручных изменений схемы
* Каждая миграция — один атомарный шаг; обратимые миграции (`up` / `down`)
* Миграции запускаются автоматически при старте сервиса или в CI перед тестами

### 16. Конфигурация и секреты

* Конфиг через **переменные окружения** (12-factor app): никакого хардкода
* Никаких секретов в коде или git-истории
* Обязательный **`.env.example`** с описанием всех переменных и их допустимых значений
* Секреты в проде — через Vault / Kubernetes Secrets / переменные CI/CD
* При старте сервис валидирует наличие обязательных переменных и падает явно, если что-то не задано

### 17. Контракт-тесты между сервисами

Consumer-Driven Contract Testing — потребитель описывает ожидания к поставщику.
Если контракт нарушен — CI падает до прода.

До завершения автоматизации: контракт-тесты описываем вручную в `/contract-tests`.

### 18. Работа с контекстом агента (CLAUDE.md)

`CLAUDE.md` — технический журнал текущей сессии разработки. Не дублирует AGENTS.md.

Что пишем в `CLAUDE.md`:

* Текущее состояние каждого модуля (`done` / `in_progress` / `todo`)
* Принятые архитектурные решения и их причины
* Открытые вопросы и блокеры
* Контекст, который нужен агенту при следующем запуске

Структура:

```
## Статус модулей
## Принятые решения
## Открытые вопросы
## Следующий шаг
```

Обновляем после каждого мержа в `main`.

### 19. Структура документов в репозитории

Карта файлов — что где лежит и кого обслуживает:

| Файл / директория | Потребитель | Назначение |
| --- | --- | --- |
| `README.md` | Инженер-потребитель + создатель | Витрина: что, зачем, API, как запустить |
| `AGENTS.md` | ИИ-агент | Неизменяемые правила разработки (этот файл) |
| `CLAUDE.md` | ИИ-агент | Изменяемый контекст текущей сессии |
| `CONTRIBUTING.md` | Мейнтейнер | Правила контрибьюции |
| `backlog.md` | Агент + разработчик | Очередь задач |
| `docs/architecture.md` | Создатель + агент | Иерархия модулей, блок-схема entry point |
| `docs/adr/` | Мейнтейнер | Архитектурные решения (ADR) |
| `api-specification/` | Потребитель + агент | OpenAPI / AsyncAPI контракты |
| `component-tests/` | Мейнтейнер + агент | Gherkin-спецификации |
| `skills/` | ИИ-агент | Процедуры в формате SKILL.md (Anthropic) — детерминированные алгоритмы для агента |
| `devlog/` | Обучающийся | Журнал разработки: промпты, решения, результаты |

НЕ дублируем: concept-уровень описания системы живёт в `ubik-life/concept`, а не в каждом сервисе. Сервис ссылается, не копирует.

### Триггеры обновления документов

| Событие | Что обновить |
| --- | --- |
| Новый эндпоинт / изменение контракта | `api-specification/`, `README.md` (таблица API + pipe) |
| Изменение entry point или новый домен | `docs/architecture.md` (блок-схема) |
| Значимый архитектурный выбор | `docs/adr/` + запись в `CLAUDE.md` (Принятые решения) |
| Merge шага | `CLAUDE.md` (Статус модулей, Следующий шаг), `backlog.md` (Done) |
| Изменение структуры репо / способа запуска | `README.md` (Структура репозитория, Запуск) |
| Завершение шага разработки | `devlog/NN-topic.md` |

### Форматы документов

- Devlog-запись: [`docs/templates/devlog.md`](docs/templates/devlog.md) — создаётся после мержа каждого шага
- ADR: [`docs/templates/adr.md`](docs/templates/adr.md) — создаётся при значимом выборе технологии, паттерна или структуры

### 20. Сценарий первой сессии агента

Когда разработчик загружает CLAUDE.md и AGENTS.md — выполни по порядку без ожидания дополнительных команд:

1. Проверь существует ли `backlog.md`:
   - **Нет** → предложи создать backlog по флоу из §9, затем предложи перейти к шагу 1: OpenAPI-спека и README
   - **Да** → прочитай backlog, найди первую задачу со статусом `todo` и предложи её выполнить

**Правила выполнения:**
- Рутинные операции (создание/редактирование файлов, обновление backlog) — выполняй без остановки
- Перед `git commit` — остановись, покажи результат, жди подтверждения
- Деструктивные действия (удаление файлов, изменение API-контракта) — явное подтверждение
- НЕ мержи в main — это делает только разработчик через PR

### Правила работы агента

- После завершения каждого шага обновлять `backlog.md`: переносить выполненные задачи в раздел `## Done`
- Обновлять документы согласно триггерной таблице §19
- Флоу завершения шага:

```
1. git checkout -b feat/<name>
2. git add <files>
3. git commit -m "type(scope): description"
4. git push -u origin feat/<name>
5. PR → merge (делает разработчик)
6. git checkout main && git pull origin main
7. git checkout -b feat/<next-name>
```

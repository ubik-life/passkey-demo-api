# CONTRIBUTING.md

Правила для тех, кто вносит изменения в сервис.

---

## Быстрый старт

1. Клонируй репозиторий
2. Настрой окружение — см. `.env.example`
3. Запусти сервис — см. `README.md → Запуск`
4. Убедись что компонентные тесты проходят — см. `component-tests/`

## Флоу разработки

Правила разработки, коммитов и веток — в `AGENTS.md`.

Коротко:
- Ветка от `main`, живёт не дольше 1–2 дней (AGENTS.md §11)
- Коммиты по Conventional Commits (AGENTS.md §12)
- Сначала тест, потом код (AGENTS.md §5)
- Изменения API — только через спеку в `api-specification/` (AGENTS.md §2)

### Именование веток

| Префикс | Когда |
|---------|-------|
| `feat/` | новая функциональность |
| `fix/` | исправление бага |
| `test/` | только тесты |
| `docs/` | только документация |
| `chore/` | конфигурация, зависимости |

## Перед отправкой PR

- [ ] Компонентные тесты проходят (`component-tests/`)
- [ ] Юнит-тесты проходят
- [ ] `api-specification/` обновлена если изменился API
- [ ] `README.md` обновлён если изменился API, стек или способ запуска
- [ ] Коммиты по Conventional Commits
- [ ] Ветка актуальна относительно `main`

## Правила code review

**Автор PR:**
- Описание PR объясняет «зачем», не «что» (что видно из diff)
- Один PR — одно логическое изменение
- Не мержи без апрува

**Ревьюер:**
- Проверяй корректность логики, не стиль (стиль — линтер)
- Если нашёл проблему — объясни почему, не только что исправить
- Апрув = готов взять ответственность за этот код

## Документация

При изменениях обновляй документы согласно триггерной таблице в `AGENTS.md §19`.

Шаблоны для новых документов — в `docs/templates/`.

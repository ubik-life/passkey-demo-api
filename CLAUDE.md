**Passkey Demo API** — Go-сервер, реализующий полный цикл авторизации без паролей (WebAuthn + JWT + SQLite). Первый кирпич ноды Ubik.

UI-часть: `git@github.com:ubik-life/passkey-demo-ui.git`

## Документирование процесса

Этот проект разрабатывается как обучающий материал. Каждая сессия работы с агентом фиксируется в `devlog/`:

```
devlog/
  00-setup.md                    ← стартовая точка: как начать проект с нуля
  00-intent.md                   ← зачем строим, API-контракт, решения по архитектуре
  01-api-contract.md             ← OpenAPI-спека
  02-gherkin.md                  ← компонентные тесты
  03-go-server.md                ← TDD-цикл: Go-сервер
  04-component-tests-skill.md    ← SKILL.md для агента (генерация компонентных тестов)
  05-failure-modes.md            ← режимы отказа: контракт, спека, карта в README
  06-component-tests-template.md ← шаблон godog в Docker, placeholder-сервис, smoke
```

### Формат каждого файла devlog

```markdown
# NN — Название шага
```

Цель — чтобы студент смог повторить весь путь шаг за шагом, работая с ИИ-агентом так же, как это делал автор. Журнал версионируется вместе с кодом.

## Принятые решения

- REST API: двухфазные POST вместо PUT для завершения регистрации/входа (см. `devlog/00-intent.md`)
- Термины `attestation` и `assertion` в URI — взяты из WebAuthn напрямую
- Trunk Based Development: ветки живут 1–2 дня
- Коммиты по Conventional Commits
- Спека в `api-specification/openapi.yaml` (OpenAPI 3.1, versioning через `/v1/`)
- Компонентные тесты: формула `N_тестов = N_эндпоинтов_API + Σ режимы отказа`. Для passkey-demo-api — 6 эндпоинтов + 2 режима SQLite = 8 сценариев. Контракт первичен: режимы отказа в OpenAPI и README, не в коде. Процедура — `skills/component-tests/SKILL.md`
- Карта режимов отказа живёт в README отдельным разделом (видимый контракт сервиса наружу), а не в AsyncAPI и не в коде адаптера
- Два различимых режима отказа SQLite: `db_locked` (503 + `Retry-After`) и `db_disk_full` (507). Правило различения: режим — отдельный, если различается хотя бы одно из четырёх — HTTP-код, заголовки, действие клиента, действие оператора. **В контракт идут только реально воспроизводимые в тестах режимы.** `db_unavailable` для встроенной SQLite — синтетический (нет «сетевой недоступности»), поэтому не выделяется; вернётся при миграции на сетевую БД. `db_timeout`/`db_corrupt` тоже отложены
- Приватный ключ Ed25519 генерируется при старте процесса и не персистится — таким образом, файловая система не становится дополнительной интеграцией с режимами отказа. TODO: вынести решение в ADR
- Раннер компонентных тестов (godog) запускается **в Docker**, не с хоста. Полная изоляция окружения раннер ↔ SUT: устраняет зависимость от локального Go-toolchain, сеть между раннером и сервисом — внутренняя docker-сеть, поведение в CI и локально совпадает до байта

# Passkey Demo API

> **Учебный проект.** Это простой MVP, созданный как демонстрация процесса разработки с ИИ-агентом. Не предназначен для использования в продакшне.

Go-сервер, реализующий полный цикл авторизации без паролей: **WebAuthn + JWT (Ed25519) + SQLite**.

Первый кирпич ноды [Ubik](https://github.com/ubik-life/concept) — открытой децентрализованной платформы для распространения знаний.

---

## Что умеет / НЕ умеет

**Умеет:**
- Регистрация пользователя по handle + биометрия (passkey)
- Вход по handle + биометрия
- Выход с инвалидацией сессии
- Проверка текущей сессии

**НЕ умеет:**
- Хранение контента (рилсы, слайды, аудио) — следующие сервисы
- Федерация с другими нодами — отдельный этап
- Мультипользовательские роли и права

| Компонент | Технология |

## API

Регистрация и вход — двухфазные: первый `POST` создаёт challenge, второй завершает процесс.

| Метод | Ресурс | Действие |
|-------|--------|----------|
| `POST` | `/v1/registrations` | Создать challenge → `201 {id, options}` |
| `POST` | `/v1/registrations/{id}/attestation` | Завершить регистрацию → JWT |
| `POST` | `/v1/sessions` | Создать challenge → `201 {id, options}` |
| `POST` | `/v1/sessions/{id}/assertion` | Завершить вход → JWT |
| `DELETE` | `/v1/sessions/current` | Выход — инвалидация refresh token |
| `GET` | `/v1/users/me` | Текущий пользователь |

Полная спецификация: [`api-specification/openapi.yaml`](api-specification/openapi.yaml)

### Как работает (pipe)

```
POST /v1/registrations
| Принимаем handle от клиента
| Генерируем challenge (32 байта, TTL 5 мин)
| Сохраняем challenge в БД
| Возвращаем {id, options} → 201

POST /v1/registrations/{id}/attestation
| Принимаем attestation от клиента
| Проверяем challenge по id (совпадение + TTL)
| Верифицируем attestation, извлекаем публичный ключ
| Сохраняем пользователя + credential в БД
| Выдаём access token (JWT Ed25519) + refresh token → 200

POST /v1/sessions
| Принимаем handle от клиента
| Ищем пользователя по handle
| Генерируем challenge (32 байта, TTL 5 мин)
| Возвращаем {id, options} → 201

POST /v1/sessions/{id}/assertion
| Принимаем assertion от клиента
| Проверяем challenge по id (совпадение + TTL)
| Верифицируем подпись публичным ключом из БД
| Создаём сессию, выдаём access token + refresh token → 200

DELETE /v1/sessions/current
| Проверяем refresh token из запроса
| Инвалидируем сессию в БД → 204
```

## Карта режимов отказа

Что сервис обещает делать наружу при отказе каждой внешней связи.
Каждая строка таблицы = один компонентный сценарий.

### Интеграции сервиса

Интеграция = выход за процесс сервиса (сеть / файловая система / IPC).

| Кандидат | Вердикт | Обоснование |
|---|---|---|
| SQLite | интеграция | Драйвер БД, файл на диске |
| WebAuthn-библиотека | модуль | Локальный Go-пакет в памяти |
| Ed25519, JWT | модуль | Чистые криптофункции в памяти |
| Приватный ключ Ed25519 | модуль | Генерируется при старте, не персистится |

**Интеграция одна — SQLite.**

### Различимые режимы отказа

| Интеграция | Режим | HTTP-ответ | `error.code` | Поведение клиента |
|---|---|---|---|---|
| SQLite | Lock contention (`SQLITE_BUSY`) | `503` + `Retry-After: 1` | `db_locked` | Повторить с backoff |
| SQLite | Диск переполнен (только write) | `507 Insufficient Storage` | `db_disk_full` | **Не** ретраить, эскалировать |

`507` отдаётся только на эндпоинтах, которые пишут в БД (все, кроме `GET /v1/users/me`).

### Чем определяется набор режимов

Режимы выбраны не по полноте теоретического перечня, а по **различию операционной реакции**:

- `db_locked` — клиент ретраит «через секунду» по `Retry-After`. Это снимает ддос-петлю клиентов, помогает SQLite разрулить writer-конфликт быстро.
- `db_disk_full` — единственный режим, где **retry бесполезен** и нужен оператор. Отдельный HTTP-код 507 поднимает алёрт без анализа тела.

Не выделено в отдельный режим:

- `db_unavailable` — для встроенной SQLite это синтетический режим. Реальные runtime-отказы — это либо `db_locked` (lock contention), либо `db_disk_full` (запись провалилась), либо непоправимый старт (служба не поднимется и контракт не работает в принципе). «Транзиентная недоступность БД» — концепция сетевых хранилищ; для embedded-файла такого нет. При миграции на сетевую БД (Postgres) добавим.
- `db_timeout` — поведение клиента такое же как при `db_locked` (retry с backoff). Если когда-нибудь захотим разделить SLA-таймауты от lock-contention — добавим.
- `db_corrupt` — событие катастрофическое, требует ручного вмешательства, для demo избыточно.
- Constraint violation (например, UNIQUE handle) — **не** режим отказа интеграции, а доменная ошибка. Отрабатывается юнитом модуля, не компонентным тестом сервиса.

**Следствие для тестов:** два различимых режима → два компонентных сценария отказа. По формуле SKILL: `6 эндпоинтов + 2 режима = 8 тестов`.
